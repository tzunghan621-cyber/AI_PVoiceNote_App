## 3. 品質：偽代碼邏輯審查

### 🔴 Critical-1：AudioRecorder 串流單位錯誤

**位置：** Phase 1-3，`AudioRecorder.start()` 方法

**問題：** 計畫使用 `self.chunk_duration = config.get("streaming.audio_chunk_duration_sec")` 作為 async generator 的 yield 間隔。依 [[data_schema#8. 設定檔]]，此值為 `600` 秒（10 分鐘）。

這意味著 AudioRecorder 要累積 **10 分鐘** 音訊才 yield 一次給 Transcriber。但 [[system_overview#4.1 即時串流架構]] 的核心承諾是「**錄音同時轉錄**」、「**即時逐字稿持續產出，UI 即時滾動顯示**」。

計畫**混淆了兩個不同概念**：

| 概念 | 用途 | 合理間隔 |
|------|------|---------|
| 音訊**暫存分段** | 每 10 分鐘切段存 WAV 到 `data/temp/`，管理磁碟資源 | 600 秒 |
| 串流**轉錄單位** | 送給 Whisper 即時轉錄，結果即時送 UI | 5~15 秒 |

**影響：** 若不修正，使用者開始錄音後要等 10 分鐘才能看到第一行逐字稿。**根本性地破壞產品「即時」定位。**

**修正建議：**
1. AudioRecorder 應以小區塊（如 5~10 秒）yield 音訊給 StreamProcessor/Transcriber
2. 同時在背景累積，每 10 分鐘寫一次暫存 WAV（`audio_chunk_duration_sec` 控制此行為）
3. 在 `config/default.yaml` 新增 `streaming.transcribe_chunk_sec`（如 10 秒），與 `audio_chunk_duration_sec` 分離
4. 或：AudioRecorder 只管串流小區塊，WAV 暫存寫入邏輯由 StreamProcessor 或獨立 buffer 負責

### 🟡 Major-1：Exporter 自動刪除音檔 vs UI 確認

**位置：** Phase 2-4，`Exporter.export()` 方法（計畫第 528-531 行）

**問題：** 計畫中 Exporter 在匯出後**直接刪除暫存音檔**。但 [[ui_spec#7. 對話框與通知]] 明確定義：

> 匯出成功 → 對話框 →「匯出完成。是否刪除原始音檔？」[刪除] [保留]

**修正建議：** Exporter 只負責匯出 Markdown，不主動刪除音檔。音檔刪除由 UI 層在使用者確認後呼叫。

### 🟡 Major-2：會議資訊對話框缺失

**位置：** Phase 4-2，`DashboardView`

**問題：** [[ui_spec#2.2]] 定義了開始錄音/匯入時的「會議資訊」快速填寫對話框（會議名稱 + 與會人員），包含 [開始] 和 [跳過] 選項。計畫的 DashboardView 偽代碼完全未涵蓋此流程。

**影響：** 使用者無法在開會前填寫會議資訊和與會人員，影響 Gemma 的摘要品質（缺少參與者上下文）。

### 🟡 Major-3：決議區塊在 Dashboard 中缺少獨立呈現

**位置：** Phase 4-2，`DashboardView._build_live_dashboard()`

**問題：** [[ui_spec#2.4]] 明確定義「決議事項獨立區塊呈現，不混在摘要文字中」。但計畫只有 `HighlightsPanel`（重點）和 `ActionsPanel`（Actions），decisions 沒有獨立處理。`SummaryResult.decisions` 在 UI 上如何呈現完全未規劃。

**修正建議：** 在 HighlightsPanel 中新增 decisions 子區塊，或將其重命名為 `SummaryPanel` 以涵蓋 highlights + decisions。

### 🟡 Major-4：會中 Action Items 保護機制不足

**位置：** Phase 4-2，`DashboardView.on_new_summary()`

**問題：** [[ui_spec#2.3]] 和 [[data_schema#5.1]] 規定「使用者手動修改的內容會被保護，AI 週期更新只影響未被手動修改的部分」。計畫對 highlights 有 `_user_edited_highlights` 檢查，但對 Action Items 只寫了 `self.actions_panel.merge(summary.action_items)` — 沒有說明如何判斷哪些 Action Items 是使用者手動修改/新增的、merge 時如何保護它們。

**修正建議：** 每個 ActionItem 需有 `user_edited: bool` 標記（或由 UI 層維護一個 edited_item_ids set），merge 時跳過已編輯項目。此涉及 data_schema，建議回報大統領評估是否需要更新 specs。

### 🔵 Minor-1：Summarizer 增量切片方式脆弱

**位置：** Phase 2-2，`Summarizer.generate()`，第 417 行

**問題：** `segments[previous_summary.covered_until:]` 用 `covered_until`（segment 全局序號）直接作為 list index。若 segment 有間隙（如某些 chunk 轉錄失敗導致跳號），切片會錯位。

**修正建議：** 改用 `[s for s in segments if s.index > previous_summary.covered_until]`。

### 🔵 Minor-2：階層式摘要僅有 stub

**位置：** Phase 2-2，`Summarizer._check_hierarchical_needed()`

**問題：** [[system_overview#4.4]] 詳細定義了階層式摘要的段落分群與合併策略。計畫只列出方法名稱，無任何邏輯。

**影響：** 低。短中期會議不需要此功能，但長期需補完。建議在 Phase 2 自測時標記為 TODO。

### 🔵 Minor-3：虛擬捲動未提及

**位置：** Phase 4-2

**問題：** [[system_overview#4.3]] 提到「UI 逐字稿：虛擬捲動，只渲染可見區域」。計畫未提及實作策略。

**影響：** 短會議無感，但 2 小時以上的會議可能造成 UI 卡頓。

## 5. 風險矩陣

### 5.1 覆蓋度（✅ 良好）

高/中/低三級分類合理，核心風險（CPU 推理慢、記憶體、格式不穩定、RAG 誤判）均已識別。應對策略具體可行。

### 5.2 遺漏的風險

| 遺漏風險 | 建議等級 | 說明 |
|----------|---------|------|
| **Whisper + Gemma 同時推理的 CPU 競爭** | 中 | 兩個模型同時跑 CPU 推理，可能互相拖慢。計畫只分別評估了記憶體，但未考慮 CPU 時間片競爭。建議：Gemma 摘要設為低優先級 thread，或確保串行（不在 Whisper 忙碌時啟動摘要）。 |
| **sounddevice 跨平台相容性** | 低 | Windows 上可能需要特定 audio backend（WASAPI/DirectSound）。建議 Phase 1 自測時記錄 Windows 環境注意事項。 |

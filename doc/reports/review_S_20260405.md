## 3. 品質：邏輯與架構審查

### 🟡 Major-1：StreamProcessor `await` 同步方法 — 生產環境 TypeError

**位置：**
- `stream_processor.py:39` — `new_segments = await self.transcriber.transcribe_chunk(audio_chunk, chunk_id)`
- `transcriber.py:22` — `def transcribe_chunk(self, ...)` — **同步方法，非 async**

**問題：**

`await` 一個非 coroutine 的普通方法，在 Python 中會拋出：
```
TypeError: object list can't be used in 'await' expression
```

測試未偵測到此 bug，因為 `test_stream_processor.py:50` 將 mock 的 `transcribe_chunk` 設為 `AsyncMock`，自動回傳 coroutine。**生產環境中使用真實 Transcriber 時會直接崩潰。**

**影響：** StreamProcessor.run() 在處理第一個音訊區塊時就會拋出 TypeError，整條 Pipeline 無法運作。

**延伸問題：** 即使修正為 `async def`，`faster_whisper.model.transcribe()` 本身是 CPU 密集的同步阻塞操作。若直接在 async event loop 中執行，會阻塞整個事件循環，導致：
1. AudioRecorder 的 `_audio_callback` 無法將資料放入 queue → 音訊丟失
2. UI 事件無法處理 → 畫面凍結

**修正建議：** 將 Whisper 轉錄放入 thread pool executor，避免阻塞 event loop：

```python
# transcriber.py — 改為 async，內部用 to_thread
import asyncio

async def transcribe_chunk(self, audio_data: np.ndarray, chunk_id: int) -> list[TranscriptSegment]:
    segments_iter, info = await asyncio.to_thread(
        self.model.transcribe, audio_data,
        language=self.language, vad_filter=True,
    )
    # segments_iter 是 generator，需在同一 thread 中消費
    # 改為先在 thread 中收集完畢
    ...
```

或者（更簡潔）在 StreamProcessor 中包裝：

```python
# stream_processor.py
new_segments = await asyncio.to_thread(
    self.transcriber.transcribe_chunk, audio_chunk, chunk_id
)
```

此方式不需改動 transcriber.py 介面，且自動將 CPU 密集操作放入 thread pool。

**測試也需同步修正：** 新增一個整合測試驗證真實（或 lightweight stub）Transcriber 與 StreamProcessor 的 async 互動，而非純 AsyncMock。

## 4. 安全性

### 4.1 路徑構造（🔵 低風險，建議改善）

| 位置 | 程式碼 | 風險 |
|------|--------|------|

| `session_manager.py:65` | `path = self.sessions_dir / f"{session.id}.json"` | Session.id 為 uuid4()，安全 |

**評估：** `term['id']` 可來自使用者手動輸入（UI 新增詞條）或批次匯入（import_yaml_batch），理論上可被注入 `../../etc/cron` 等路徑。但本 App 為純本地 Desktop 應用，使用者即為管理者，實際風險極低。

**建議（可延後）：** 在 `add_term` / `import_yaml_batch` 時驗證 `id` 格式（如 `re.match(r'^[a-z0-9_]+$', id)`），符合 data_schema 定義的「英文小寫 + 底線」。

### 4.2 其他安全項 — ✅

- **YAML 使用 `safe_load`**：ConfigManager、KnowledgeBase 均使用 `yaml.safe_load`，無 arbitrary code execution 風險

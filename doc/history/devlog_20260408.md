# 開發日誌 — 2026-04-08

## V Phase 持續踩雷：Flet 0.84.0 API breaking changes

### 今日狀態

- 模型選型已最終確定：預設 `gemma4:e2b`（甲方實測 E4B 跑不動）
- 碼農完成預設值修正（commit `3b0b459`）
- V Phase 重啟，實驗者繼續驗證

### 模型決策回顧

甲方回報：「我用 ollama 直接掛 gemma4 e2b 是可以直接在他上面跑，對話是 ok 的」「我也確定 e4b 他跑不動」

決策：
- 預設改為 `gemma4:e2b`
- E4B 改為高階升級選項（≥32GB RAM 或有 GPU）
- E2B 對 Surface Pro 9 才是合理配置

### 連續四個 Flet API breaking change

V Phase 進行中陸續發現 Flet 0.84.0 多個 API 變動：

| # | API | 觸發 | 狀態 |
|---|-----|------|------|
| 1 | Page setter（具體待查） | 啟動初期 | 已修 |
| 2 | OutlinedButton.color | UI 渲染 | 已修 |
| 3 | FilePicker.on_result（callback 廢除） | 點匯入音檔 | 已修（commit `d8186ba`） |
| 4 | Tab.text / Tab.content（架構變動） | 進入 live 模式 | **發現中** |

### Bug #3 修復細節

碼農修復 FilePicker：
- 改為新 API：`pick_files()` / `save_file()` 直接 await 回傳
- 移除 `on_result` callback、`overlay.append()`、預先 `page.update()`
- 影響 3 處：dashboard_view（匯入音檔、匯出 Markdown）、terms_view（匯入詞條 YAML）
- 127 tests 通過

### Bug #4 — 觸發策略轉變

實驗者發現第 4 個 API 變動後，提出關鍵建議：

> 「兩天連續發現 4 個 Flet 0.84.0 API breaking change。建議大統領請 Researcher 系統性掃過 Flet 0.84.0 升級指南，一次找出所有不相容處，避免逐個踩雷。」

**大統領決策：採納。**

逐個踩雷的問題：
- 每次測試都可能撞牆，效率極低
- 浪費實驗者時間
- 可能還有未知的隱性 bug

新策略：
1. **停止逐個修**
2. 派**研究者**系統性掃 Flet 0.84.0 全部 breaking changes
3. 對照本專案 `app/ui/` 程式碼，列出所有受影響檔案和行號
4. 派**碼農**一次修完所有不相容處
5. 再請**實驗者**續驗

### 教訓：版本相容性

這次踩雷的根因是：
- **specs 階段未鎖定 Flet 版本**：`pyproject.toml` 用 `flet>=...` 而非 `flet==0.84.0`
- **依賴更新時未做 migration check**：Flet 0.84.0 明顯有大量 breaking change
- **TDD 測試覆蓋不到 UI 渲染**：core 模組 106 tests 全綠，但 UI 層的 API 不相容靠人工冒煙才能發現

未來要做的：
- 重要依賴鎖定具體版本
- 升級依賴前先讀 changelog
- UI 層至少做一次 smoke test 確認啟動

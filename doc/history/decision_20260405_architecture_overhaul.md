# 決策紀錄 — 架構翻修

## 背景

Specs v1 設計為批次處理架構（錄音 → 等待 → 審閱）。甲方在討論中釐清了真實使用情境，觸發全面翻修。

## 甲方原話

> 「開會時就把他開在右側，或會議室大螢幕，大家一邊開會他一邊錄音，一段小時間後大家的逐字稿會正確，也會有會議結論、行動項目。開會時一個階段會切換出來讓大家看對不對。」

> 「他自己就有一個知識庫，但我們平常是用你(claude code)，把我的知識庫連過來。他自己也是一個小型的知識庫。」

## 決策內容

### 1. Pipeline：批次 → 串流

| 舊 | 新 | 原因 |
|---|---|------|
| 錄完才開始轉錄 | 錄音同時即時轉錄 + 校正 | 會議中需要即時看到逐字稿 |
| 全部完成後一次產生摘要 | 每 3~5 分鐘週期性增量更新 | 會議中需要階段性確認結論和 Action Items |
| 處理中顯示等待進度條 | 不需要等待畫面，直接在儀表板上即時填充 | 使用者開會時不應該看到空白等待畫面 |

### 2. UI 模態：三階段 → 兩模態

| 舊（三階段） | 新（兩模態） |
|-------------|-------------|
| 2.2 錄音中（空畫面+音量指示器） | 2.3 即時儀表板（三區塊唯讀，即時更新） |
| 2.3 處理中（進度條等待） | （不需要，串流處理無等待） |
| 2.4 工作區（會後審閱編輯） | 2.4 會後編輯模式（同畫面切為可編輯） |

### 3. 知識庫：外部索引 → App 自有

| 舊 | 新 | 原因 |
|---|---|------|
| App 是 Obsidian 知識庫的索引工具 | App 自有知識庫，外部同步充實 | 甲方定位：App 本身是智能產品，不是附屬工具 |
| 知識庫路徑指向 Obsidian | 知識庫存在 App 內（data/terms + data/chroma） | 獨立運作，不依賴外部 |
| 由使用者手動維護 | 三種來源：Obsidian 同步 + 手動 + 自動建議 | 多管道充實知識 |

### 4. 產品定位演進

```
v0: 語音會議摘要工具
v1: 個人會議指揮中心（可編輯工作區）
v2: 會議即時智能儀表板（會中即時 + 會後編輯 + 自有知識庫）
```

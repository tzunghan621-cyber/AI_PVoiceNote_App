# 開發日誌 — 2026-04-04

## 專案啟動

甲方提出需求，經多輪討論確認以下方向：

### 需求確認

- 直接做成品，不做 MVP
- 本地端推理（Gemma 4），不走雲端 API
- Desktop App（Flet 框架）
- 輸出留在 app 內審閱，甲方手動決定匯出
- 匯出資訊完整，交由知識庫 agent 後處理
- 原始音檔匯出完成後刪除
- 回饋紀錄存在專案中
- 輸入支援麥克風錄音 + 音檔匯入

### 產品定位演進

- 初始：語音會議摘要工具
- 調整：**個人會議指揮中心**（可編輯工作區，持續迭代進化）

### 工作區三區塊

甲方要求將產出分為三個可編輯區塊：
1. **逐字稿** — 可標記校正回饋
2. **會議重點** — AI 產生，可直接編輯修正
3. **Action Items** — 結構化待辦，含負責人/期限

### 長錄音處理

討論了長時間錄音的記憶體與效能問題，加入分段處理策略：
- 錄音每 10 分鐘切段
- 轉錄 + 校正逐段串流處理
- 超長逐字稿採階層式摘要

### 已完成

- 建立專案目錄結構（`doc/` 文件體系 + `data/` 資料目錄）
- 起草三份 specs：[[system_overview]]、[[data_schema]]、[[ui_spec]]
- 所有文件加入 `[[wikilink]]` 跨引用，符合 Obsidian 格式
- 建立 `CLAUDE.md` 專案指引

### 競品研究完成

研究者完成競品分析報告（[[competitive_analysis_20260404]]），覆蓋 8 個競品。

確認三大差異化成立：
1. RAG 知識庫校正 — 市面僅有簡單字典（Otter/Fireflies），無 RAG
2. 三區塊工作區 — 所有競品皆為分頁或上下排列
3. 回饋驅動優化循環 — 完全獨有

### 工作區佈局決策

甲方使用情境：半邊螢幕多工、外接螢幕。決定採用三段式響應佈局：
- ≥ 1400px：三欄並排
- 960 ~ 1399px：兩欄（逐字稿 + 右側分頁切換重點/Actions）
- < 960px：單欄分頁

採納競品啟發：
- Granola 的 AI vs 使用者編輯視覺區分 → 用於會議重點區
- 逐字稿校正處可「一鍵新增詞條」降低回饋門檻
- 欄寬可拖曳調整，偏好值記憶

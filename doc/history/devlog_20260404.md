# 開發日誌 — 2026-04-04

## 專案啟動

甲方帶著已有的知識庫文件來開始專案：
- 專案規劃文件：`極客坊_語音會議摘要筆記.md`
- 技術學習筆記：`學習筆記_語音會議摘要.md`
- Gemma 4 參考：`學習筆記_Gemma.md`

大統領閱讀完畢後，與甲方進入多輪需求討論。

---

### 第一輪：基本方向確認

甲方明確表態：
- **「我沒有要做 MVP，我要直接做出成品」**
- 不走雲端 API，直接用 Gemma 4 本地推理

大統領提出技術選型供確認：

| 決策 | 結果 | 甲方態度 |
|------|------|----------|
| 摘要引擎 | Gemma 4 4B（Ollama） | 甲方直接選本地推理 |
| 向量資料庫 | ChromaDB | 「這部份我不懂，直接依你的」 |
| Embedding | paraphrase-multilingual-MiniLM-L12-v2 | 同上 |
| Desktop 框架 | Flet | 同上 |

> **觀察：** 甲方不介入技術選型細節，授權大統領決策。

---

### 第二輪：三階段使用模式

甲方主動定義了產品的三個使用階段：
1. **養庫** — 平常叫 Claude Code 把知識庫重點建進來
2. **使用 + 回饋** — 程式使用後做回饋
3. **優化** — 根據回饋調整知識庫

> **關鍵洞察：** 甲方從一開始就把這個產品定位成「持續進化的系統」，不是一次設定好就用的工具。

---

### 第三輪：資料進出

大統領主動詢問資料流定義，甲方逐項確認：

**輸入：**
- 麥克風即時錄音 + 音檔匯入（WAV/MP3/M4A）

**輸出：**
- 「輸出的東西應該不能直接用，由我來決定匯出」
- 「匯出應該還是儘可能資訊完整，再由知識庫的 agent 來處理」
- 「檔案太肥了，我們用完就丟」→ 原始音檔匯出後刪除

**回饋：**
- 「回饋記錄要存在我們專案中」→ `data/feedback/`

---

### 第四輪：開發規範

甲方要求先讀開發規範（`AI協作開發規範.md`），遵循 P-S-C-V 流程。

大統領詢問角色分配，甲方回答：
- **「Director 是你，我是甲方」**
- 甲方不是 Director，不參與流程管理，只負責需求和簽核

> **決策紀錄：** 見 [[decision_20260404_roles]]

### 第五輪：團隊命名

甲方為角色取代號：
- 甲方 = 甲方
- Director = **大統領**
- Researcher = **研究者**
- Builder / Reviewer / Verifier = 待命名

甲方要求團隊名冊要 doc 化：**「你的名冊在哪裡，我們專案要完全 doc 化，不依賴 memory 與對話上下文」**

> **重要原則：** 所有資訊必須落在 doc 裡，不能只存在對話 context 或 memory 中。

---

### 第六輪：Specs 起草與簽核

大統領起草三份 specs：

- [[data_schema]] — 資料結構、詞條/回饋/匯出格式

---

### 第七輪：長錄音問題

甲方提出：**「如果錄音錄很久，會爆炸，你要怎麼處理」**

大統領加入分段處理策略：
- 錄音每 10 分鐘切段，不累積在記憶體
- Whisper 逐段串流處理
- 超長逐字稿採階層式摘要（分段摘要 → 合併）

---

### 第八輪：工作區與產品進化

甲方提出兩個重大方向：
1. **「要規劃工作區，有時候有錯就可以開始改了。再分成逐字、會議重點與 Action 解析」** → 三區塊工作區概念
2. **「我們這個專案會持續進化，將會是我的開會指揮中心」** → 產品定位從工具升級為指揮中心

甲方隨即要求先做競品研究再細化，避免過早鑽細節。

---

### 第九輪：競品研究

大統領指派研究者（CLI-4）執行競品調查。產出報告：[[competitive_analysis_20260404]]

**研究結果摘要：**
- 覆蓋 8 個競品（Otter.ai、Fireflies.ai、tl;dv、Krisp、Notta、Granola、Meetily、Talat）
- 確認三大差異化成立：RAG 校正、三區塊並排、回饋驅動優化
- Meetily（開源）技術棧最接近但缺 RAG
- Talat 同為本地 + Obsidian 但僅 macOS

**採納的設計啟發：**
- Granola：AI vs 使用者編輯用不同樣式區分
- 三欄寬度可拖曳，窄視窗自動折疊
- 校正處「一鍵新增詞條」降低回饋門檻

---

### 第十輪：響應式佈局

甲方說明使用習慣：**「我可能習慣把他拉到另一個螢幕，或是用半邊螢幕同時多工其它工作」**

大統領決定三段式響應佈局：
- ≥ 1400px：三欄並排
- 960 ~ 1399px：兩欄（逐字稿 + 右側分頁）
- < 960px：單欄分頁

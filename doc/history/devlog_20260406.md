# 開發日誌 — 2026-04-06

## V Phase 啟動與模型選型危機

### 實驗者 V Phase 啟動

- 甲方安裝 Ollama for Windows
- 實驗者開始環境驗證 + 冒煙測試
- 冒煙測試第一次失敗 → 實驗者發現 bug 修正
- 卡在模型下載：`gemma4:4b` 不存在

### 模型名稱錯誤（大統領失職）

**根因**：技術選型（4/4）時，大統領假設 Ollama tag 為 `gemma4:4b`，未實際驗證。此錯誤散布至全專案 doc + code，經歷 P Phase、S Phase、兩輪 Review 都沒人發現，直到 V Phase 實際 `ollama pull` 才爆出。

**正確名稱**：`gemma4:e4b`（E = Efficiency 版本，4.5B 有效參數 / 8B 總參數）

**修正範圍**：
- 大統領：specs（system_overview、data_schema、ui_spec）、CLAUDE.md、ollama_guide
- 碼農：config、summarizer、main_view、5 個 test 檔案

### 甲方體驗反饋

甲方實測後認為 Surface Pro 9 跑起來可能不夠力，提出：
- 需要更多模型選項
- 要做可攜式（整個資料夾帶著走）
- 計畫連知識庫給太極門成員用

### 四層級方案（短暫存在 → 被推翻）

大統領提出四層級方案（gemma3:1b / 3:4b / 4:e2b / 4:e4b）。
碼農實作完成，審察者 Review Pass。

但隨後研究者完成 Gemma 4 完整分析報告（[[gemma4_model_analysis_20260406]]），揭示：
- E4B 載入 RAM 只需 6-7GB（非 9.6GB 檔案大小）
- 品質超越 Gemma 3 27B
- 原生 JSON/function calling

甲方看完報告，明確表態：**「不，我沒要用 gemma3 系列的模型」**

> 決策紀錄：[[decision_20260406_model_selection]]

### 最終選型

| 預設 | 降級 | 限制 |
|------|------|------|
| `gemma4:e4b` | `gemma4:e2b` | `num_ctx: 8192`, `temperature: 0.3` |

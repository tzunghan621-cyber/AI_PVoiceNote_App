# 開發日誌 — 2026-04-05

## 架構翻修：從批次工具到即時儀表板

甲方明確了產品的真正使用情境，觸發整體架構翻修。

### 觸發原因

甲方描述的場景：
> 「開會時就把他開在右側，或會議室大螢幕，大家一邊開會他一邊錄音，一段小時間後大家的逐字稿會正確，也會有會議結論、行動項目。開會時一個階段會切換出來讓大家看對不對。」

這與原設計（錄音 → 等待 → 審閱）根本不同。

### 架構變更

| 面向 | 舊（v1） | 新（v2） |
|------|---------|---------|
| Pipeline | 批次處理（錄完才跑） | 串流處理（錄音同時跑） |
| 摘要 | 全部完成後一次產生 | 每 3~5 分鐘週期性增量更新 |
| UI 模態 | 錄音 → 等待 → 審閱 | 會中（即時儀表板）→ 會後（編輯工作區） |
| 會中行為 | 空畫面等錄完 | 三區塊即時唯讀更新 |
| 知識庫 | 外部 Obsidian 的索引工具 | App 自有知識庫，外部同步充實 |
| 產品定位 | 會議摘要工具 | 會議即時智能儀表板 |

### 知識庫架構重構

甲方明確：App 本身是一個有知識的智能產品，不是依附 Obsidian 的索引工具。

```
App 自有知識庫（data/terms/ + ChromaDB）
  ├── 來源 1：Claude Code 從 Obsidian 同步（obsidian_sync）
  ├── 來源 2：使用者手動新增（manual）
  └── 來源 3：App 從高頻遺漏自動建議（auto_suggest）
```

### 討論過程

#### 即時處理的發現

大統領提出「錄音中畫面太空」的問題，甲方進一步釐清：
- 不是「會後才打開看」，而是「開會時就投影在大螢幕」
- 不只是自己用，與會者都能看到
- 每 3~5 分鐘大家會切出來看結論對不對

#### 知識庫定位轉變

甲方原話：「他自己就有一個知識庫，但我們平常是用你(claude code)，把我的知識庫連過來。他自己也是一個小型的知識庫。」

這改變了知識庫的根本定位：
- 舊：App 是 Obsidian 的索引前端
- 新：App 是獨立智能體，Claude Code 是它的「進修老師」

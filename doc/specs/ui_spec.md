## 2. 會議頁（主頁）

### 2.1 初始狀態（無進行中 Session）

```
┌─────────────────────────────────────┐
│                                     │
│     🎙️ [開始錄音]    📁 [匯入音檔]   │
│                                     │
│         🎤 [測試麥克風]              │
│                                     │
│─────────────────────────────────────│
│  歷史紀錄                            │
│                                     │
│  📋 2026-04-04 14:30 — 週會（已匯出） │
│  📋 2026-04-03 10:00 — 站會（待審閱） │
│  📋 2026-04-02 15:00 — 客戶會議      │
│                                     │
└─────────────────────────────────────┘
```

**功能：**
- [開始錄音] → 進入即時儀表板（2.3）
- [匯入音檔] → 開啟檔案選擇器（WAV/MP3/M4A），匯入後進入儀表板，逐步填充內容
- **[測試麥克風]** → 進入 5 秒 mic test 模式（見 §2.5），不建立 session、不寫 WAV，倒數結束自動回 idle
- 歷史紀錄列表 → 點擊進入會後編輯模式（2.4）

### 2.2 會議資訊

開始錄音或匯入音檔時，快速填寫對話框：

```
┌──────────────────────────────┐
│  會議資訊（可稍後修改）        │
│                              │
│  名稱：[2026-04-05 週會   ]  │
│                              │
│  與會人員：                   │
│  [John, Mary, +新增...    ]  │
│                              │
│          [開始]  [跳過]      │
└──────────────────────────────┘
```

- 名稱預設值：日期 + 時間
- 與會人員：可輸入多人，逗號分隔或逐個新增
- 可全部跳過，稍後在儀表板修改
- Gemma 在摘要週期中會從逐字稿推斷與會者，自動補充（標記為 AI 推斷，見 [[data_schema#5. Session（單次會議）]]）

### 2.3 即時儀表板（會中模態）

這是**會議進行中的主畫面**。設計目標：投影在大螢幕或放在筆電側邊，所有與會者可見。

#### 響應式佈局

**寬視窗（≥ 1400px）— 三欄並排**

```
│  📜 逐字稿    │  💡 會議重點      │  ✅ Action Items     │
```

#### 儀表板行為（會中）

| 區塊 | 即時行為 | 互動 |
|------|---------|------|
| **逐字稿** | 即時滾動，新 segment 出現在底部，校正標記即時顯示 | 唯讀，自動捲動（使用者往上捲時暫停自動捲動，點「回到最新」恢復） |
| **會議重點** | 每 3~5 分鐘自動更新，重點累積、決議累積 | **可編輯** — 助理可即時修正；已編輯的區塊不被 AI 週期更新覆蓋 |
| **Action Items** | 每 3~5 分鐘自動更新，項目累積 | **可編輯** — 可即時修改內容、補負責人/期限、新增/刪除項目 |

> **會中設計原則：** 逐字稿唯讀（保持資料完整性），會議重點和 Action Items 可編輯（助理可即時修正 AI 產出）。使用者手動修改的內容會被保護，AI 週期更新只影響未被手動修改的部分。

#### 頂部列（會中）

```
🔴 錄音中  00:23:15   🎤 ▮▮▮▮▮▯▯▯▯▯   [週會 2026-04-05]  👥 John, Mary +2    [⏹ 停止錄音]
                     ↑ Mic Live 音量條（見 §2.5）
```

- 錄音指示燈 + 時長
- **Mic Live 音量條**（§2.5）— 即時反映麥克風訊號強度，甲方可立即發現收音異常
- 會議名稱（可點擊修改）
- 與會人員摘要（點擊可展開編輯，AI 推斷的人員會標記）
- 停止錄音按鈕

#### 狀態列（會中）

```
🟢 Ollama │ Gemma4:E4B │ 42 詞條 │ 上次摘要: 00:21:30 │ 下次摘要: ~00:24:30
```

比一般狀態列多顯示：上次/下次摘要更新時間，讓使用者知道「它有在跑」。

### 2.4 會後編輯模式

停止錄音 → 最終摘要產生完成 → 自動切換為會後編輯模式。同一個三區塊畫面，但從唯讀變成可編輯。

#### 佈局

與即時儀表板相同的三段式響應佈局（三欄/兩欄/單欄），差異在於：

```
┌────────────────────────────────────────────────────────┐
│  ✅ 會議結束  01:02:15   週會 2026-04-05               │
├──────────────┬──────────────────┬──────────────────────┤
│  📜 逐字稿    │  💡 會議重點      │  ✅ Action Items     │
│              │                  │                      │
│  可標記校正    │  ✏️ 可編輯全文    │  ✏️ 可編輯/新增/     │
│  回饋         │                  │     刪除/勾選完成     │
│              │  AI 原文灰色      │                      │
│              │  使用者改的白色    │  含負責人/期限/       │
│              │                  │  優先級               │
│              │                  │                      │
├──────────────┴──────────────────┴──────────────────────┤
│  [匯出 Markdown]    [提交回饋]    [刪除]                │
└────────────────────────────────────────────────────────┘
```

#### 三區塊（會後可編輯）

| 區塊 | 內容 | 可編輯 | 資料來源 |
|------|------|--------|----------|
| **逐字稿** | 帶時間戳的校正後逐字稿 | 校正回饋（正確/錯誤/遺漏），不開放全文編輯 | [[data_schema#3. 校正結果]] |
| **會議重點** | AI 最終摘要 + 決議 | 全文可編輯，AI vs 使用者修改以不同樣式區分 | [[data_schema#4. 摘要結果]] + [[data_schema#5.1 使用者編輯]] |
| **Action Items** | 結構化待辦 | 表單式編輯各欄位，可新增/刪除/勾選完成 | [[data_schema#4. 摘要結果]] |

#### 逐字稿校正互動

```
[00:05:12] 我們決定用 [Gemma 4]⚡ 來做本地推理
                       ↑ 校正：原始「寶石四」→「Gemma 4」
                         [✅ 正確] [❌ 錯誤] [+ 新增詞條]
```

- 校正處以琥珀色高亮，hover/點擊顯示校正詳情
- 每個校正可直接標記回饋（正確/錯誤）
- 可回報「遺漏」：選取文字 → 右鍵 → 「回報遺漏校正」
- 遺漏回報可「一鍵新增詞條」至 App 知識庫，降低回饋門檻（參考 [[competitive_analysis_20260404#7.4 RAG 知識庫 — 獨特競爭優勢]]）

#### 會議重點編輯

- AI 原始產出以灰色/淡色顯示
- 使用者編輯/新增的內容以白色/亮色區分（參考 Granola 的人機協作概念，見 [[competitive_analysis_20260404#2.6 Granola]]）
- 決議事項獨立區塊呈現，不混在摘要文字中
- 編輯內容獨立存放（[[data_schema#5.1 使用者編輯]]），不覆蓋 AI 原始產出

#### 底部操作

- [匯出 Markdown] → 匯出時包含三區塊完整內容（格式見 [[data_schema#7. 匯出格式（Markdown）]]）→ 匯出後提示刪除音檔
- [提交回饋] → 批次儲存校正回饋（格式見 [[data_schema#6. 回饋紀錄（Feedback）]]）
- [刪除] → 刪除整個 Session（需二次確認）

### 2.5 Mic Live 指示器

**目的：** 即時反映麥克風訊號強度，解決「錄完半小時才發現收不到聲音」的痛點（V Phase 第五輪實機觀察到甲方 Surface Pro 9 峰值 -52 dBFS，VAD 全過濾導致 segments 空）。

**視覺元件：**

```
🎤 ▮▮▮▮▮▯▯▯▯▯    ← 10 格音量條
```

顏色分級（由 AudioRecorder 每 200ms 量測當前 RMS 的 dBFS）：

| 狀態 | dBFS 範圍 | 顏色 | 含義 |
|------|----------|------|------|
| 🔇 靜音/太小 | peak < -40 | 深灰（閃爍提示） | **警訊** — 可能麥克風沒插/增益不足/被靜音 |
| 🟢 正常 | -30 ~ -12 | 綠色 | 適合轉錄 |
| 🟡 大聲 | -12 ~ -3 | 黃色 | 訊號強但未爆 |
| 🔴 爆音 | > -3 | 紅色 | 可能削波/失真，建議降低增益 |

**出現時機：**
- 會中模態（§2.3）：常駐於頂部列
- Mic Test 模式（§2.1 [測試麥克風]）：畫面中央大顯示 + 5 秒倒數
- idle / review 模式：不顯示

**Mic Test 模式（閒置預覽）：**

```
┌─────────────────────────────────────┐
│                                     │
│         🎤 測試麥克風 (倒數 4s)       │
│                                     │
│         ▮▮▮▮▮▯▯▯▯▯                 │
│         peak: -28 dBFS (正常)        │
│                                     │
│         [取消測試]                    │
│                                     │
└─────────────────────────────────────┘
```

- 持續 5 秒（可設定 `ui.mic_indicator.test_duration_sec`）
- **不建立 Session、不寫 WAV 暫存**（純即時量測）
- 倒數結束或 [取消測試] 自動回 idle
- 用途：會議前先確認麥克風設定、太極門成員自檢硬體

**資料來源：**
- AudioRecorder 新增 API `get_current_level() -> float`（回傳 dBFS, -80 ~ 0）
- 背景維持 rolling buffer（最近 200ms）計算 RMS，不影響轉錄主路徑
- UI 以 `page.run_task` + 定時器 poll，更新 ProgressBar 寬度與顏色

**規格欄位（data_schema #8 config）：**

```yaml
ui:
  mic_indicator:
    poll_interval_ms: 200          # UI 更新頻率
    threshold_silent_dbfs: -40     # 以下顯示警訊
    threshold_normal_dbfs: -30     # 以上為正常範圍下限
    threshold_loud_dbfs: -12       # 以上為大聲
    threshold_clipping_dbfs: -3    # 以上為爆音
    test_duration_sec: 5           # Mic Test 模式倒數秒數
```

**無障礙考量：**
- 視障使用者：條旁顯示文字（如「peak: -28 dBFS（正常）」）
- 靜音警訊 > 3 秒持續時，SnackBar 提示「麥克風訊號極弱，請檢查設定」

## 3. 詞條管理頁

管理 App 自有知識庫的詞條。詳見 [[system_overview#5. 知識庫架構]]。

```
┌─────────────────────────────────────┐
│  知識詞條管理          [+ 新增] [匯入] │
│─────────────────────────────────────│
│  🔍 搜尋詞條...        篩選：[全部 ▼] │
│─────────────────────────────────────│
│                                     │
│  詞條        別名       來源  命中/成功│
│  ──────────────────────────────────│
│  Gemma 4    寶石四...   🔗    15/12 │
│  Claude     克勞德...   ✋     8/7  │
│  RAG        拉格...     🔗     3/1  │
│  ChromaDB   chroma      💡     0/0  │
│                                     │
│─────────────────────────────────────│
│  共 42 筆詞條 │ 平均成功率 78%        │
└─────────────────────────────────────┘
```

**來源圖示：** 🔗 = `obsidian_sync`（Claude Code 同步）、✋ = `manual`（手動新增）、💡 = `auto_suggest`（App 自動建議）

**功能：**
- 搜尋：即時過濾詞條
- 篩選：依分類、依來源、依效能（高命中/低命中/零命中）
- 新增：開啟詞條編輯表單
- 匯入：批次匯入 YAML 檔案（詞條格式見 [[data_schema#1. 知識詞條（Term）]]）
- 點擊詞條列 → 展開/開啟詳情編輯
- 命中/成功欄位：顯示 `correction_count / success_count`

### 詞條編輯表單

```
┌─────────────────────────────────────┐
│  編輯詞條                            │
│                                     │

│  統計：命中 15 次 │ 校正 12 次        │
│        成功 10 次 │ 失敗 2 次         │
│                                     │
│           [儲存]  [取消]  [刪除]      │
└─────────────────────────────────────┘
```

**功能：**
- 整體統計：成功率、回饋數、Session 數
- 需要關注區：自動分析出需要處理的詞條
  - 低成功率 → 建議補充別名或調整
  - 零命中 → 建議移除（避免向量空間汙染）
  - 高頻遺漏 → 建議新增詞條（可一鍵新增至 App 知識庫）
- 最近 Session 回饋列表

## 8. 視覺風格
